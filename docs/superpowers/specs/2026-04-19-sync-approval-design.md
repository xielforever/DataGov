# 数据建模高危操作（同步发布）审批流程设计

## 1. 需求背景与目标
数据模型向底层物理数据源同步（建表/删表）属于高危操作。直接在开发页面执行可能导致生产环境的严重事故。因此，需在现有系统中引入“工单化”的审批机制，将开发与发布权限分离。

本设计采用 **方案A：集中式审批中心**。

## 2. 核心状态流转设计
目前数据模型具有 `status: 'draft' | 'published' | 'offline'`。
为了支持审批流，我们扩展一个模型中间状态：`approving`（审批中）。
同时，扩展一个“同步状态”的中间状态：`sync_approving`（同步审批中）。

由于用户的核心诉求是**“同步物理表”时拦截**，因此核心改造点在于 `syncStatus` 或 `status` 联动：
* **普通开发者流程**：
  1. 正常编辑模型并发布 (`published`)，此时物理表还未创建。
  2. 点击【同步至数据库】，系统不直接弹出高危删除确认，而是弹出【提交同步申请】弹窗，填写申请原因。
  3. 提交后，模型同步状态变为 `sync_approving`（不可再点击同步或编辑）。
* **管理员审批流程**：
  1. 在独立的【审批中心】查看工单详情（包含模型基础信息和将要执行的 DDL）。
  2. 点击【同意并执行同步】 -> 系统在后台调用底层数据库接口执行建表/删表 -> 成功后状态变为 `synced`，并记录日志。
  3. 点击【驳回】 -> 状态退回 `unsynced`。

## 3. UI 界面变更

### 3.1 数据建模页面 (DataModeling.tsx)
* **操作列按钮变更**：当状态为 `published` 且未同步时，“同步至数据库”按钮变更为“申请同步”。如果处于 `sync_approving` 状态，则按钮置灰并显示“审批中”。
* **弹窗变更**：原有的“红色高危确认框”改造为“填写工单申请单”，说明此次同步的改动范围，并提交给指定的数据域负责人/DBA。

### 3.2 新增审批中心页面 (ApprovalCenter.tsx)
* **入口位置**：在左侧全局主导航（Sidebar）中，将【审批中心】放置为**倒数第二个一级菜单**（位于“数据服务”和“系统管理”之间）。
* **二级菜单设计**：
  为保持对各业务模块审批的通用适配性（如标准定义、码值管理、数据建模等），审批中心下设以下**基于个人视角**的二级菜单：
  1. `my-todos`: 【我的待办】 (统一展示需要当前用户审批的所有模块的工单)
  2. `my-applies`: 【我发起的】 (统一展示当前用户提交的申请)
  3. `my-processed`: 【已处理】 (统一展示当前用户已审批过的记录)
* **页面结构 (通用适配性)**：
  * **列表页**：各二级菜单页面采用统一的表格结构。提供顶部的“所属模块（如模型发布、标准定义、码值变更）”下拉框筛选功能。表格列包括：工单编号、所属模块、申请事项（如“同步表 ods_xxx”）、申请人、申请时间、审批状态。
  * **动态详情抽屉 (Drawer)**：点击某条工单，侧边滑出详细内容。抽屉内部内容区域根据 `approval.moduleType` 动态渲染不同的详情组件。例如：
    * `moduleType === 'data_model'`：展示 DDL 建表语句及受影响数据源。
    * `moduleType === 'data_standard'`：展示标准定义的核心属性对比（如旧标准 vs 新标准）。
    * `moduleType === 'code_value'`：展示码值字典的增删改差异项。

## 4. API 及 Mock 数据改造 (handlers.ts / data.ts)
* **模型接口**：
  * 修改 `syncModelToDb` 接口语义，从“直接同步”变为“提交同步申请”。
* **工单审批接口 (新增)**：
  * `fetchApprovals(type?: string)`: 获取审批列表，支持按视角（待办、发起、已处理）及模块筛选。
  * `handleApproval(id, action)`: 管理员审核工单。根据工单的 `moduleType` 执行不同的后置回调。对于模型同步，如果同意，则触发建表逻辑及日志记录。
* **工单数据集 (通用化扩展)**：
  * `mockApprovals` 数组的 `Approval` 接口必须设计为高度通用的结构：
    ```typescript
    interface Approval {
      id: string;
      moduleType: 'data_model' | 'data_standard' | 'code_value'; // 核心扩展
      title: string;       // 申请事项概要，如 "申请同步物理表: ods_user"
      applicant: string;
      applyTime: string;
      reason: string;
      status: 'pending' | 'approved' | 'rejected';
      processTime?: string;
      processor?: string;
      payload: any;        // 泛型存储业务数据（如模型ID，标准定义的变更JSON等），供动态详情组件消费
    }
    ```

## 5. 阶段实施计划
1. **第一阶段**：调整现有的“同步”按钮逻辑为“提交申请”流程，更新 `DataModeling.tsx` 和相关接口。
2. **第二阶段**：搭建新的路由 `/approvals` 及 `ApprovalCenter.tsx` 页面。
3. **第三阶段**：将原本写在 DataModeling 中的“确认执行并写日志”的 Mock 逻辑，迁移至“审批通过”的接口里。

## 6. 异常处理与降级
* 若工单审批通过但最终同步底层数据源失败（例如网络超时），则模型状态变为 `failed`，工单本身状态为“已完成（但同步失败）”，需开发者查看日志并重新发起。
